
# Cypher: Сервис для интеграции с КриптоПро



Сервис поставляется в виде Docker-контейнера, который включает в себя:
*   **КриптоПро CSP 5.0** и **КриптоПро ЭЦП SDK 2.0 (cades)** для выполнения криптоопераций.
*   **Stunnel** для создания защищенного TLS-туннеля до внешнего ресурса.
*   **FastAPI-приложение** для предоставления REST API.
*   **Набор интеграционных тестов** на `pytest` для проверки целостности системы.

## Основные возможности

*   **Полное формирование SOAP-конвертов:** Генерация готовых к отправке, подписанных по стандарту WS-Security SOAP-конвертов через один вызов API.
*   **Управление сертификатами:** Установка корневых и личных сертификатов при старте контейнера, а также API для их просмотра.
*   **Прозрачное туннелирование:** Встроенный и настроенный Stunnel для работы с ГИИС.
*   **Разделение сред:** Поддержка режимов `dev` и `prod` с разным уровнем логирования и автоматическим запуском тестов в `dev`-режиме.
*   **Самодиагностика:** Информативные тесты, которые при падении подсказывают возможную причину неисправности.

## 1. Подготовка к запуску

Перед запуском необходимо подготовить два каталога в корне проекта: `dist` и `assets`.

### Шаг 1: Подготовка дистрибутивов (`dist/`)

Этот каталог должен содержать установочные файлы КриптоПро. Вы должны получить их с [официального сайта КриптоПро](https://www.cryptopro.ru/downloads), так как они требуют лицензии.

Структура каталога `dist`:
```
dist/
├── cades-linux-amd64.tar.gz  # КриптоПро ЭЦП SDK 2.0
├── linux-amd64_deb.tgz       # КриптоПро CSP 5.0
└── pycades.zip               # Библиотека-обертка для Python
```

### Шаг 2: Подготовка пользовательских данных (`assets/`)

Этот каталог содержит ваши сертификаты и ключи. 

#### 2.1 Личный сертификат и ключ (`assets/certificates/my/`)

Система ожидает найти здесь один zip-архив с именем `bundle.zip`. Этот архив должен содержать **сертификат** и **контейнер с закрытым ключом**.

**Как создать `bundle.zip`:**
1.  Найдите ваш контейнер с закрытым ключом. Обычно это папка с шестью файлами (`header.key`, `masks.key` и т.д.).
2.  Экспортируйте ваш личный сертификат в файл (например, `user.cer`).
3.  Создайте zip-архив `bundle.zip`, который будет содержать:
    *   Файл сертификата `user.cer`.
    *   Папку с контейнером ключа.

Пример структуры `bundle.zip`:
```
bundle.zip
├── user.cer
└── keys/
    └── 1a2b3c4d.000/
        ├── header.key
        ├── masks.key
        ├── masks2.key
        ├── name.key
        ├── primary.key
        └── primary2.key
```

#### 2.2 Корневые сертификаты (`assets/certificates/roots/`)

Сюда необходимо поместить все корневые и промежуточные сертификаты удостоверяющих центров в форматах `.cer`, `.crt` или `.p7b`. Эти сертификаты будут автоматически установлены в доверенное хранилище `Root` при старте контейнера.

#### 2.3 Сертификат для Stunnel (`assets/stunnel/`)

Здесь должен находиться **клиентский сертификат в формате `.cer`**, который будет использоваться Stunnel для аутентификации при установке TLS-соединения.
```
assets/stunnel/
└── client.cer
```
**Важно:** Обычно это тот же самый сертификат, что и ваш личный (`user.cer` из `bundle.zip`). Просто скопируйте его сюда.

### Шаг 3: Конфигурация (`.env`)

Создайте файл `.env` в корне проекта на основе примера ниже и укажите ваши параметры.

```dotenv
# .env

# Режим работы приложения:
# dev - включает отладочное логирование и авто-запуск тестов при старте.
# prod - стандартный режим работы.
APP_MODE=dev

# Адрес и порт удаленного хоста, до которого будет строиться туннель
STUNNEL_REMOTE_HOST=api.dmdk.ru
STUNNEL_REMOTE_PORT=443

# Локальный порт, на котором Stunnel будет принимать соединения
STUNNEL_ACCEPT_PORT=1500
```

## 3. API Документация

Сервис предоставляет REST API на порту `8000`. Интерактивная документация доступна по адресу `http://localhost:8000/docs`.

### Основной эндпоинт

#### `POST /api/v1/signature/dmdk/create-signed-envelope`

Принимает XML с бизнес-данными, полностью формирует и подписывает SOAP-конверт по стандарту WS-Security и возвращает его в виде готовой к отправке XML-строки. Это основной и рекомендуемый способ использования сервиса.

**Запрос:**
```json
{
  "request_data_xml": "<ns:SendGetGlossaryRequest xmlns:ns='urn://xsd.dmdk.goznak.ru/exchange/3.0'><ns:TestMessage>true</ns:TestMessage><ns:OGRN>1234567890123</ns:OGRN><ns:RequestData id='body'><ns:type>GEMSTONE_COLOR_TYPE</ns:type><ns:page>1</ns:page><ns:size>25</ns:size></ns:RequestData></ns:SendGetGlossaryRequest>"
}
```




### Служебные эндпоинты

#### `GET /health`

Проверка доступности сервиса.
*   **Ответ:** `{"status": "ok"}`

#### `GET /api/v1/certificates/client`

Возвращает подробную информацию о сертификате, используемом для подписи.
*   **Успешный ответ (200 OK):**
    ```json
    {
      "issuer": "CN=Test CA",
      "subject": "CN=Your Company",
      "valid_from": "2023-01-01T10:00:00",
      "valid_to": "2024-01-01T10:00:00",
      "thumbprint": "...",
      "has_private_key": true,
      "days_left": 300,
      "is_active": true
    }
    ```
*   **Ошибка (404 Not Found):** Если активный сертификат с ключом не найден.

#### `POST /api/v1/signature/verify`

Проверяет присоединенную подпись в формате PKCS#7. Может использоваться для проверки подписи в ответе от ГИИС.

**Запрос:**
```json
{
  "signed_data": "MII..." // Подписанные данные в формате Base64
}
```

**Успешный ответ (200 OK):**
```json
{
  "is_valid": true,
  "message": "Signature verified successfully."
}
```

## 4. Как это работает

### Логика запуска (`entrypoint.sh`)
1.  **Установка корневых сертификатов:** Скрипт ищет сертификаты в `/assets/certificates/roots` и устанавливает их. Если каталог пуст, шаг пропускается.
2.  **Установка личного сертификата:** Скрипт ищет `/assets/certificates/my/bundle.zip` и устанавливает из него сертификат и закрытый ключ. Если файл отсутствует, шаг пропускается.
3.  **Запуск тестов (в `dev`-режиме):** Если `APP_MODE=dev`, запускаются внутренние тесты `pytest` для проверки работоспособности крипто-ядра.
4.  **Запуск сервисов:** Запускается `supervisor`, который управляет процессами FastAPI и Stunnel.

### Логика подписи (`SoapSignerFactory`)
Для повышения производительности при старте приложения создается единственный экземпляр `SoapSignerFactory`, который кэширует информацию о сертификате для подписи. При вызове эндпоинта `create-signed-envelope` фабрика выполняет следующие шаги по стандарту WS-Security:
1.  Оборачивает бизнес-данные в `<soap:Body>` с идентификатором `wsu:Id="body"`.
2.  Вычисляет каноническую форму и хэш (ГОСТ 34.11-2012) от `<soap:Body>`.
3.  Формирует элемент `<ds:SignedInfo>`, содержащий хэш тела запроса.
4.  Подписывает `<ds:SignedInfo>` с помощью закрытого ключа (ГОСТ 34.10-2012).
5.  Собирает полный заголовок `<wsse:Security>`, включающий `<ds:SignedInfo>`, значение подписи `<ds:SignatureValue>` и сам сертификат в `<wsse:BinarySecurityToken>`.
6.  Собирает финальный SOAP-конверт, объединяя заголовок и тело запроса.
